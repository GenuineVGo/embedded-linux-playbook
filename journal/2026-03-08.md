# Journal — 2026-03-08

## Objective
- Yocto RPi4 : reproduire les invariants Buildroot (SSH par clé, mDNS `.local`, hostname fixe) dans l'image Yocto via kas + meta-vincent.

## What worked
- `ssh-keys.bb` : recette Yocto créée dans `meta-vincent/recipes-security/ssh-keys/` — installe `authorized_keys` root dans `/home/root/.ssh/` (perms `700`/`600`). Note : Poky place le home root en `/home/root/`, différent de Buildroot (`/root/.ssh`). Preuve : `ls -l /home/root/.ssh/authorized_keys` → `-rw------- root root` + contenu clé OK.
- `layer.conf` corrigé : patterns `BBFILES` étendus pour couvrir 1, 2 et 3 niveaux de profondeur → la nouvelle recette est bien découverte par BitBake (point clé).
- `BBFILE_PATTERN_vincent` passé de `=` à `:=` (vérifié, laissé en l'état — pas critique ici, `=` fonctionne aussi pour `BBFILE_PATTERN_*`).
- `rpi4.yml` (kas) mis à jour :
  - `debug-tweaks` ajouté dans `EXTRA_IMAGE_FEATURES` (root sans mot de passe en dev).
  - `ssh-server-dropbear` retiré de `EXTRA_IMAGE_FEATURES` et mis dans `IMAGE_FEATURES` — règle : `EXTRA_IMAGE_FEATURES` ne doit pas contenir de packages, seulement des features.
  - **`libnss-mdns`** (et non `nss-mdns`) + `avahi-daemon avahi-utils dbus` → stack mDNS `.local` sur le LAN. Le nom exact `libnss-mdns` était le blocage build.
  - Hostname fixé : `hostname:pn-base-files = "rpi4-yocto"` (mécanisme Yocto natif via `base-files`).
  - `IMAGE_INSTALL:append = " ssh-keys"` → recette SSH incluse dans l'image.
- **Build Yocto RPi4 validé** (Poky 5.0.16).
- **SSH par clé OK** : `ssh root@rpi4-yocto.local` depuis Windows → login root sans mot de passe, clé ed25519 acceptée.
- **mDNS `.local` OK** : `rpi4-yocto.local` résolu et accessible via `ssh root@rpi4-yocto.local` (IP dynamique, non utilisée directement).
- **Hostname OK** : `rpi4-yocto` visible sur UART et dans le prompt SSH.
- **Réseau OK** : eth0 DHCP `192.168.137.198/24`, GW `192.168.137.1`, DNS `192.168.137.1`.
- **UART OK** : boot complet visible via picocom (`115200 8N1`), login root possible.
- `dmesg | grep -i error` → pas d'erreurs bloquantes au boot.

## What broke
- `BBFILES` initial dans `layer.conf` ne couvrait que 2 niveaux (`recipes-*/*/*.bb`) → la recette `recipes-security/ssh-keys/ssh-keys.bb` n'était pas détectée par BitBake.
- `ssh-server-dropbear` dans `EXTRA_IMAGE_FEATURES` : incorrect — `EXTRA_IMAGE_FEATURES` est réservé aux features, pas aux packages. Déplacé dans `IMAGE_FEATURES` (ajout via kas/`local.conf` reste valide).
- `systemctl` absent sur l'image : Poky utilise **sysvinit** par défaut, pas systemd → `systemctl status avahi-daemon` ne fonctionne pas. Avahi est actif (mDNS prouve son fonctionnement) mais non observable via systemctl.
- IP DHCP dynamique (`192.168.137.198`) — voulu : Yocto utilise DHCP, Buildroot utilise IP fixe (`192.168.137.6` via `S40network`). Le mDNS `.local` assure l'accessibilité sans IP fixe.
- Host key RSA connue par IP (`known_hosts:11: 192.168.137.198`) → warning SSH "host known by other name" au premier connect via `.local` (résolu par `yes`).

## Fix / command / reference

### layer.conf : extension des patterns BBFILES
```bitbake
BBFILES += "${LAYERDIR}/recipes-*/*.bb \
            ${LAYERDIR}/recipes-*/*.bbappend \
            ${LAYERDIR}/recipes-*/*/*.bb \
            ${LAYERDIR}/recipes-*/*/*.bbappend \
            ${LAYERDIR}/recipes-*/*/*/*.bb \
            ${LAYERDIR}/recipes-*/*/*/*.bbappend"
```

### Recette ssh-keys.bb
```bitbake
do_install() {
    install -d -m 0700 ${D}/home/root/.ssh
    install -m 0600 ${WORKDIR}/authorized_keys ${D}/home/root/.ssh/authorized_keys
}
FILES:${PN} += "/home/root/.ssh/authorized_keys /home/root/.ssh"
```

### Hostname via Yocto (base-files)
```bitbake
hostname:pn-base-files = "rpi4-yocto"
```

### IMAGE_FEATURES / EXTRA_IMAGE_FEATURES / IMAGE_INSTALL
- `IMAGE_FEATURES` / `EXTRA_IMAGE_FEATURES` : pour les **features** (ex. `ssh-server-dropbear`, `debug-tweaks`). Les deux peuvent être alimentés depuis kas/`local.conf`.
- `IMAGE_INSTALL` : pour les **packages** (ex. `avahi-daemon`, `libnss-mdns`, `ssh-keys`). Ne jamais mettre des packages comme `avahi-daemon` ou `libnss-mdns` dans `EXTRA_IMAGE_FEATURES`.

### Stack mDNS (Yocto)
- Packages : `avahi-daemon avahi-utils libnss-mdns dbus`
- Attention : le paquet Yocto s'appelle **`libnss-mdns`** (et non `nss-mdns`) — c'était le blocage build.
- Miroir du setup résolu côté CM3588/Buildroot (cf. journal 2026-03-04).
